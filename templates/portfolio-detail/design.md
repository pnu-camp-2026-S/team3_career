# 1번 상세기술 포트폴리오 디자인 기준

## 1. 템플릿 목적

상세기술 포트폴리오는 사용자의 프로젝트 경험을 과정, 기여도, 결과 중심으로 깊게 정리하는 PPT 형식이다.

이 템플릿은 단순 이력 요약이 아니라, 채용 담당자가 사용자의 문제 해결 흐름과 실제 기여를 이해할 수 있도록 구성한다. 프로젝트 경험이 여러 개일 경우 같은 구조를 반복해 뒤에 붙인다.

## 2. 전체 구조

상세기술 포트폴리오는 `표지 1장 + 경험별 3장 반복` 구조를 사용한다.

| 구분 | 페이지 | 역할 |
|---|---|---|
| 공통 | 지원자 소개 | 지원자 정체성, 희망 직무, 핵심 역량, 연락처를 보여준다. |
| 경험별 반복 | 경험 개요 | 프로젝트명, 문제 상황, 목표, 역할, 기간, 사용 기술을 정리한다. |
| 경험별 반복 | 과정 및 기여도 | 진행 과정, 내가 맡은 일, 구현 흐름, 협업 방식, 문제 해결 과정을 보여준다. |
| 경험별 반복 | 결과 및 회고 | 완성 결과, 사용자 가치, 한계점, 개선 방향, 배운 점을 정리한다. |

경험이 늘어나면 `경험 개요 → 과정 및 기여도 → 결과 및 회고` 3장 묶음을 계속 추가한다.

## 3. 디자인 방향

- 출력 형식은 16:9 가로형 PPT를 기본으로 한다.
- 전체 분위기는 흰 배경, 얇은 구분선, 연한 블루 포인트, 넓은 여백을 사용한다.
- 제목은 크게, 본문은 짧게, 정보는 카드와 표 형태로 정리한다.
- 페이지 상단에는 작은 섹션 라벨을 둔다. 예: `Portfolio`, `Project 01`, `Result`.
- 실제 프로젝트 화면, 목업, 작업 이미지, 구조도 등 시각 자료를 반드시 포함한다.
- 사진은 장식이 아니라 이해를 돕는 근거 자료로 사용한다.
- 색상은 블랙, 그레이, 연한 블루를 중심으로 사용하고 과한 장식은 피한다.
- 한 페이지에는 하나의 핵심 메시지만 담는다.

## 4. 페이지별 구성

### 4.1 지원자 소개

#### 포함 내용

- 이름
- 희망 직무
- 한 줄 소개
- 핵심 역량 3~5개
- 연락처 또는 이메일
- 프로필 사진 또는 대표 이미지

#### 레이아웃 기준

- 상단 중앙에 `Portfolio` 라벨을 배치한다.
- 중앙에는 지원자를 설명하는 큰 문장을 배치한다.
- 하단에는 연락처 정보 바를 둔다.
- 프로필 사진이나 대표 이미지는 과하지 않게 보조 요소로 사용한다.

### 4.2 경험 개요

#### 포함 내용

- 프로젝트명
- 프로젝트 한 줄 설명
- 문제 상황
- 목표 사용자
- 프로젝트 목표
- 본인 역할
- 기간
- 사용 기술
- 프로젝트 대표 이미지 또는 서비스 화면

#### 레이아웃 기준

- 상단에 `Project 01`처럼 경험 번호를 표시한다.
- 제목은 `경험 1: 프로젝트를 설명하는 결론 한 줄` 형식으로 작성한다.
- 본문은 `문제 상황`, `목표 사용자`, `핵심 목표` 같은 3개 내외의 카드로 구성한다.
- 프로젝트 대표 이미지나 화면 캡처를 함께 배치한다.

### 4.3 과정 및 기여도

#### 포함 내용

- 진행 과정
- 내가 맡은 역할
- 구현한 기능
- 데이터 또는 화면 흐름
- 협업 방식
- 어려웠던 점과 해결 방법
- 작업 화면, 흐름도, 구조도, 목업 이미지

#### 레이아웃 기준

- 진행 과정은 `요구 분석 → 화면 설계 → 기능 구현 → 테스트·수정`처럼 흐름으로 보여준다.
- 본인 기여도는 팀 전체 결과와 구분해 작성한다.
- 이미지가 있는 경우 왼쪽에는 대표 이미지, 오른쪽에는 설명 영역을 배치한다.
- 문제 해결 과정은 `문제 → 판단 → 해결` 순서로 짧게 정리한다.

### 4.4 결과 및 회고

#### 포함 내용

- 구현 결과
- 사용자에게 제공한 가치
- 완성된 화면 또는 산출물 이미지
- 한계점
- 개선 방향
- 배운 점

#### 레이아웃 기준

- 결과 화면, 최종 산출물, KPI 카드 등 결과를 보여주는 이미지를 포함한다.
- 수치가 없는 경우 임의로 만들지 않고 정성적 결과로 표현한다.
- 하단에는 `배운 점`을 한 문장으로 강조한다.
- 회고는 변명이나 감상문이 아니라 다음 개선 방향과 연결되게 작성한다.

## 5. AI 작성 원칙

- 업로드 자료, 승인된 Wiki, 사용자가 입력한 활동 기록처럼 확인 가능한 근거만 사용한다.
- 없는 성과, 수치, 역할, 기술, 프로젝트명은 만들지 않는다.
- 사용자가 승인하지 않은 초안 내용은 최종 포트폴리오에 사용하지 않는다.
- 경험별로 문제, 과정, 기여, 결과가 자연스럽게 이어지도록 정리한다.
- 본인 기여와 팀 결과를 명확히 구분한다.
- 긴 문단보다 제목, 짧은 설명, 카드형 문장을 우선한다.
- 사진이나 이미지는 프로젝트 이해에 필요한 위치에 배치한다.

## 6. 피해야 할 것

- 프로젝트를 단순 소개문처럼 길게 나열하는 것
- 이미지 없이 텍스트만으로 구성하는 것
- 근거 없는 성과 수치나 역할을 만들어내는 것
- 기술 스택만 나열하고 실제 구현 흐름을 설명하지 않는 것
- 본인 기여 없이 팀 전체 결과만 강조하는 것
- 경험이 늘어날 때마다 서로 다른 구조로 페이지를 만드는 것
