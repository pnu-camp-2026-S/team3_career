# AGENTS.md

이 문서는 Myfitfolio/CareerFit 프로젝트에서 작업하는 AI 에이전트와 개발자가 반드시 따라야 할 최상위 기준이다.  
세부 기능 기준은 `docs/` 폴더의 기능별 문서에서 관리한다.

모든 새 문서, 주석, 화면 문구, 테스트 설명은 특별한 사유가 없는 한 한국어로 작성한다.

## 1. 프로젝트 한 줄 정의

Myfitfolio는 취업을 준비하는 대학생이 흩어진 활동 자료를 업로드하면, AI가 근거 기반 LLM Wiki를 만들고, 사용자가 승인한 내용만으로 직무 맞춤형 포트폴리오와 활동 추천을 제공하는 서비스이다.

## 2. 대상 사용자

- 취업을 준비하는 대학교 3~4학년 학생
- 보고서, 코드, 발표자료, 실험 결과, 자기소개서, 이력서 등은 가지고 있지만 취업용 포트폴리오 형식으로 정리하지 못한 사용자
- 프로그래밍이나 AI에 익숙하지 않아 화면 흐름과 안내 문구가 쉬워야 하는 사용자

## 3. 가장 중요한 제품 원칙

1. AI는 근거 없는 경험, 성과, 수치, 역할을 만들면 안 된다.
2. 업로드 자료, 작업 기록, 사용자 대화처럼 확인 가능한 근거를 바탕으로만 정리한다.
3. 사용자가 승인하기 전의 Wiki 내용은 초안으로만 취급한다.
4. 포트폴리오는 승인된 Wiki 내용만 사용한다.
5. 추천 기능은 단순 목록이 아니라 `왜 이 활동이 목표 직무와 포트폴리오에 도움이 되는지`를 설명해야 한다.
6. 같은 경험이라도 목표 직무에 따라 강조점이 달라질 수 있음을 고려한다.
7. AI가 수정안을 바로 원본에 반영하지 말고, 사용자가 확인 후 적용할 수 있게 한다.

## 4. 현재 기술 스택

- 앱 프레임워크: Next.js, React
- 화면 자산: 기존 HTML, CSS, JavaScript를 유지하고 Next.js 라우트에서 렌더링한다.
- API: Next.js Route Handler
- AI API: OpenAI API
- 인증 보조: `bcryptjs`
- 데이터 저장: 전환 단계에서는 `localStorage`/`sessionStorage`를 유지하고, 이후 Supabase 기준으로 설계한다.
- 환경 변수: `key.env`와 `dotenv`
- 테스트: Node.js 기본 `assert` 기반 정적 구조 테스트
- 배포 예정: Vercel

## 5. 실행 및 검증 명령

프로젝트 루트에서 실행한다.

```bash
npm install
npm run dev
npm run build
npm test
```

로컬 개발 서버 주소:

```text
http://localhost:3000/
```

`key.env`, `.env`, `.env.*`에는 API 키가 들어가므로 커밋하지 않는다.

## 6. 저장소 구조 기준

```text
.
├─ app/                  # Next.js App Router, API Route, 정적 자산 라우트
├─ lib/                  # Next 전환 유틸
├─ docs/                 # 기능별 프로젝트 기준 문서
├─ html/                 # 현재 서비스 화면의 원본 HTML
├─ css/                  # 공통 및 화면별 CSS
├─ js/                   # 공통 및 화면별 브라우저 JavaScript
├─ tests/                # 정적 구조 테스트
├─ package.json
├─ README.md
└─ AGENTS.md
```

Next.js 전환 단계에서는 기존 화면과 동작 보존을 우선한다.  
DB/Supabase 연동은 전환 완료 이후 페이지별 이슈와 PR로 진행한다.

## 8. 공통 UI 개발 기준

1. 주요 화면은 공유 상단 내비게이션을 사용한다.
2. 브랜드 표기는 `Myfitfolio`를 기본으로 한다.
3. 사용자가 반복해서 쓰는 업무형 서비스이므로, 화면은 과도한 장식보다 명확한 정보 구조를 우선한다.
4. 버튼, 입력창, 카드, 탭의 문구는 사용자가 지금 무엇을 할 수 있는지 바로 이해할 수 있게 작성한다.
5. 화면 문구는 한국어로 작성한다.
6. 사용자가 실수했을 때는 실패 상태와 다음 행동을 함께 보여준다.
7. AI 결과는 사용자가 이해하고 수정할 수 있어야 한다.

## 9. 코드 작업 핵심 규칙

1. 기존 화면 이름과 사용자 흐름을 우선 유지한다.
2. Next.js 전환 단계에서는 기존 HTML/CSS/JS와 localStorage 기반 동작을 유지한다.
3. 활성 HTML 화면은 `html/`, 스타일은 `css/`, 브라우저 스크립트는 `js/`에 유지한다.
4. Next.js 라우팅과 API는 `app/`에서 담당한다.
5. 새 기능을 추가하면 관련 정적 구조 테스트도 함께 갱신한다.
6. UI 상태는 임시 구현 단계에서 `localStorage`를 사용할 수 있지만, 실제 서비스 기준에서는 Supabase 저장을 고려한다.
7. 사용자가 승인하지 않은 AI 결과를 최종 데이터처럼 저장하지 않는다.

## 10. 작업 완료 전 확인 목록

- `npm test`가 통과하는가?
- `npm run build`가 통과하는가?
- 새 화면 문구가 한국어인가?
- 개인정보나 API 키가 노출되지 않았는가?
- 위키의 핵심 원칙인 근거 기반, 사용자 승인 기반, 직무 맞춤 기준을 지켰는가?
- 포트폴리오 생성 또는 추천 기능에서 추천 이유와 근거가 보이는가?
- 파일 순서, 상태, 변경 이력이 저장 또는 저장 가능 구조로 반영되었는가?
- 삭제된 파일이 휴지통에서 확인, 복구, 영구 삭제 가능한가?
- DB/Supabase 연동이 필요한 기능을 Next 전환 PR에 섞지 않았는가?
