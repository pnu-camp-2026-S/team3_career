# 자료 입력 및 파일관리 기준

자료 입력 페이지의 목적은 사용자가 프로젝트별 자료를 업로드하고, AI가 요약·분류·인덱싱할 수 있도록 정리하는 것이다.

## 지원해야 할 입력 방식

- 버튼 클릭 업로드
- 드래그 앤 드롭 업로드
- AI 대화창 기반 Markdown 파일 생성
- GitHub 연동 확장용 버튼과 임시 동기화 상태 표시

## 지원 파일 예시

```text
.pdf
.ppt
.pptx
.doc
.docx
.md
.txt
.csv
.xlsx
.png
.jpg
.jpeg
```

## 파일 업로드 후 필요한 처리

1. 파일 저장
2. 파일 메타데이터 생성
3. AI 분석 및 파일 내용 추출 즉시 시작
4. 파일별 핵심 내용 요약 생성
5. `summary.md` 갱신
6. `index.json` 갱신
7. `log.md` 갱신
8. 분석 결과 상태 저장

## 프로젝트 폴더와 세부 폴더 기준

- 메인 페이지의 프로젝트 폴더와 파일 관리 탭의 프로젝트 폴더는 같은 데이터를 사용해야 한다.
- 메인 페이지에서는 프로젝트 폴더 목록을 간단히 보여 주고, 파일 관리 탭에서는 같은 프로젝트 폴더의 세부 폴더와 자료 목록을 자세히 보여 준다.
- 메인 페이지에서 프로젝트 폴더를 클릭하면 파일 관리 탭으로 이동하고, 선택한 프로젝트가 열린 상태로 보여야 한다.
- 파일 관리 탭에서는 프로젝트별로 폴더를 생성하고, 각 프로젝트 폴더 안에 자료를 추가하는 방식을 사용한다.
- 프로젝트 폴더 안에는 `발표자료`, `보고서`, `데이터`, `코드`, `기획서`, `제출 자료`처럼 자료 성격에 따른 세부 폴더를 만들 수 있어야 한다.
- 폴더 생성 시 활동 유형(개인 프로젝트/팀 프로젝트/공모전 등)을 고르면, 유형에 맞는 세부 폴더가 함께 생성된다(#137-2). 세부 폴더 구성은 `js/folder-store.js`의 `SUBFOLDER_TEMPLATES` 기본 템플릿을 따르며, 확정된 자료구조 템플릿이 오면 이 값을 교체한다.
- 세부 폴더를 클릭하면 해당 세부 폴더에 들어 있는 자료만 열려서 보여야 한다. 파일은 프로젝트 폴더의 세부 폴더에 소속되며, 업로드도 선택한 세부 폴더로 들어간다.
- 진행중 프로젝트는 `완료로 이동`으로 완료 그룹으로 옮길 수 있고, 세부 폴더와 자료가 함께 이동한다(#137-1).
- AI 정리는 개별 파일 단위가 아니라 프로젝트 단위(`프로젝트 정리`)로 실행하고, 프로젝트별 `대화로 내용 추가하기`로 현재 세부 폴더에 자료를 추가한다(#137-3/4/5).
- 파일은 반드시 하나의 프로젝트 폴더에 속해야 하며, 가능하면 하나의 세부 폴더에도 속하도록 관리한다.
- 프로젝트 폴더, 세부 폴더, 파일 목록은 동기화 가능한 하나의 데이터 구조로 관리한다.
- `js/folder-store.js`가 폴더/파일 UI 상태를 `localStorage` 기반으로 관리하며, 메인 화면과 파일 관리 화면이 모두 이 모듈을 통해서만 폴더/파일 상태를 읽고 쓴다. 실제 파일 원본과 메타데이터는 Supabase Storage와 `activity_files` 테이블에 저장한다. 폴더/파일 데이터 구조는 [데이터 구조 및 API 기준](data-model.md)을 참고한다.
- 파일 관리 탭에서 `폴더 추가`로 프로젝트 폴더를 만들면 `js/folder-store.js`에 실제 폴더 데이터가 생성되어 저장되고, 메인 사이드바에도 같은 이름으로 즉시 반영된다. 메인 사이드바는 폴더 이름을 요약 카드로만 보여 주고(자료 개수 숫자는 표시하지 않음), 자료 추가·수정은 파일 관리 탭에서만 한다.
- 파일 업로드와 삭제는 파일 관리 탭에서 `/api/activity-files` API를 통해 서버(Supabase)에 저장·삭제하고, 서버 요청이 실패하면 `localStorage` 상태로 폴백한다. 메인 화면은 이 API에서 파일 목록을 읽기만 한다.
- 프로젝트별 GitHub 레포 연결 정보는 폴더 데이터의 `github` 필드에 폴더 단위로 저장한다([GitHub 임시 연동 기준](github.md) 참고).

## 파일 순서와 우선순위

- 파일 순서는 반드시 백엔드 또는 영속 저장소에 저장한다.
- 파일 순서는 포트폴리오 작성 우선순위로 사용한다.
- `order`는 화면 표시 순서이고, `priority`는 AI 분석과 포트폴리오 생성 우선순위이다.

## 삭제와 휴지통 기준

- 삭제는 우선 soft delete 방식으로 처리하고, 삭제된 파일은 휴지통에서 확인할 수 있어야 한다.
- 휴지통에는 삭제된 파일명, 원래 프로젝트 폴더, 원래 세부 폴더, 삭제 시각, 삭제한 사용자, 분석 상태를 표시한다.
- 휴지통의 파일은 복구하거나 영구 삭제할 수 있어야 한다.
- 복구 시 파일은 원래 프로젝트 폴더와 세부 폴더 위치로 돌아가야 한다.
- 원래 폴더가 사라진 경우에는 사용자가 복구 위치를 다시 선택할 수 있게 한다.
- 영구 삭제는 사용자 확인 팝업을 거친 뒤 처리한다.
- 휴지통에 있는 파일은 AI 분석, `summary.md` 갱신, 포트폴리오 생성, 활동 추천 근거에서 제외한다.
- 파일 삭제, 복구, 영구 삭제 이력은 `log.md`에 기록한다.

## 저장 파일 기준

- 폴더 내부 파일 목록은 `index.json`에 저장한다.
- 전체 요약은 `summary.md`에 저장한다.
- 변경 이력은 `log.md`에 저장한다.
- 자료가 없는 경우 AI 대화로 `activity-summary.md` 같은 Markdown 파일을 생성해 저장할 수 있어야 한다.

## 상태값 기준

```text
uploading: 업로드 중
uploaded: 업로드 완료
failed: 업로드 실패
deleted: 삭제됨

pending: 분석 시작 전 임시 상태 또는 재시도 대기
analyzing: 분석 중
completed: 분석 완료
failed: 분석 실패

draft: 임시 저장
saved: 저장 완료
sync_pending: GitHub 동기화 대기
synced: GitHub 동기화 완료
sync_failed: GitHub 동기화 실패
```
