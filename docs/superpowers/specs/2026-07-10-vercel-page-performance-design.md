# Vercel 페이지 로딩 성능 개선 설계

## 1. 목적

Vercel에 배포된 Myfitfolio의 로그인 화면(`/`, `/index.html`, `/login.html`)과 메인 화면(`/main.html`)을 포함한 모든 사용자 화면의 초기 표시 속도를 개선한다. 기존 HTML, CSS, JavaScript 기반 화면과 사용자 흐름은 유지하고, 데이터 생성·수정·삭제 방식은 변경하지 않는다.

## 2. 확인한 병목

현재 프로덕션 빌드와 로컬 프로덕션 서버를 조사한 결과 다음 병목을 확인했다.

1. `/css/*`와 `/js/*` 요청이 정적 파일이 아니라 동적 Route Handler로 처리된다. 이 구조는 Vercel에서 정적 CDN 대신 서버 함수가 실행될 수 있어 최초 요청과 콜드 스타트 지연을 만든다.
2. `LegacyNavigation`이 첫 화면 진입 직후 11개 레거시 경로를 모두 미리 불러온다. 사용자가 방문하지 않을 페이지의 HTML과 React Server Component 데이터까지 요청한다.
3. `LegacyScripts`가 화면 스크립트를 `cache: 'no-store'`로 하나씩 내려받아 순차 실행한다. 따라서 브라우저 캐시를 활용하지 못하고, 뒤쪽 스크립트는 앞쪽 다운로드가 끝날 때까지 기다린다.
4. 로그인 화면의 `index.css`와 `login.css`는 내용 없이 `auth.css`를 `@import`한다. 브라우저가 두 번째 CSS를 확인한 뒤에야 실제 인증 화면 CSS를 요청하는 직렬 요청이 발생한다.
5. 메인 화면 초기화에서 폴더, 활동 파일, 프로필, 저장된 종합 분석 결과를 대부분 차례대로 요청한다. 서로 독립적인 요청까지 기다리므로 화면 완성 시간이 길어진다.
6. 메인 화면의 튜토리얼 이미지는 처음부터 HTML에 포함되어 있다. 튜토리얼을 열지 않는 사용자에게도 이미지 요청이 발생할 수 있다.
7. 활동 추천 화면은 글꼴과 아이콘을 위해 외부 CDN CSS 두 개를 렌더링 차단 방식으로 불러온다. 실제 외부 아이콘 사용은 이전·다음 화살표 두 개뿐이다.
8. 각 화면의 JavaScript 크기는 최대 약 73KB이며, 화면 진입 후에야 동적 Route Handler를 통해 다운로드되므로 실행 시작이 늦어진다.

로컬 프로덕션 서버의 기존 구조에서는 `/css/*`와 `/js/*` 개별 요청이 반복 측정에서 약 45~69ms 걸렸고, 응답은 `max-age=0, must-revalidate`로 매번 재검증되었다. 실제 Vercel 환경에서는 네트워크 왕복과 콜드 스타트에 따라 지연이 더 커질 수 있다.

## 3. 설계 원칙

- 첫 화면에 필요한 자원만 우선 전달한다.
- 서로 의존하지 않는 다운로드와 API 요청은 동시에 시작한다.
- 읽기 요청만 안전한 범위에서 캐시하고, 생성·수정·삭제 요청은 캐시하지 않는다.
- 일부 부가 데이터 요청이 실패해도 이미 표시할 수 있는 화면은 가리지 않는다.
- 화면 문구, 기능, 데이터 의미와 사용자 승인 기반 제품 원칙은 변경하지 않는다.
- DB 구조 변경이나 Supabase 마이그레이션은 이번 작업에 포함하지 않는다.

## 4. 구성 요소별 설계

### 4.1 CSS·JavaScript를 정적 CDN 자원으로 전환

`app/css/[...file]/route.js`와 `app/js/[...file]/route.js`에 각 디렉터리의 파일 목록을 반환하는 `generateStaticParams`를 추가한다. Route Handler는 요청 헤더나 파일 수정 시각에 의존하지 않고 빌드 시 파일 내용을 반환하도록 바꾼다.

- `dynamic = 'force-static'`과 `dynamicParams = false`를 사용한다.
- 빌드 결과에서 두 경로가 동적(`ƒ`)이 아니라 정적 생성 경로로 표시되어야 한다.
- 브라우저는 배포 변경을 확인할 수 있고 Vercel CDN은 재사용할 수 있도록 `public, max-age=0, s-maxage=31536000, must-revalidate` 정책을 사용한다.
- 원본 CSS와 JavaScript는 프로젝트 규칙에 따라 기존 `css/`, `js/` 디렉터리에 유지한다.
- 경로 이탈 방지 검사는 유지하여 허용된 디렉터리 밖의 파일을 읽지 못하게 한다.

### 4.2 필요한 화면만 의도 기반으로 미리 불러오기

첫 진입 시 모든 레거시 경로를 순회하는 사전 로딩을 제거한다. 대신 사용자가 내부 링크에 마우스를 올리거나 키보드 포커스를 이동했을 때 해당 경로만 한 번 미리 불러온다.

- 동일 경로는 한 번만 사전 로딩한다.
- 외부 링크, API 경로, 다운로드 링크에는 적용하지 않는다.
- 클릭 시 기존 Next.js 소프트 내비게이션은 유지한다.
- 사전 로딩 실패는 현재 화면이나 실제 클릭 이동을 막지 않는다.

### 4.3 레거시 스크립트 병렬 다운로드와 캐시 사용

페이지에 선언된 외부 스크립트 다운로드를 모두 동시에 시작하되, 실행은 HTML에 선언된 순서를 그대로 지킨다. 공통 스크립트가 먼저 실행되어야 하는 기존 의존성을 보존하기 위함이다.

- `cache: 'no-store'`를 제거하여 브라우저와 CDN 캐시를 사용한다.
- 인라인 스크립트는 네트워크 요청 없이 기존처럼 즉시 준비한다.
- 페이지가 바뀌면 아직 실행하지 않은 결과를 무시하고 등록한 전역 이벤트 리스너를 정리한다.
- 특정 스크립트가 실패하면 오류를 기록하되 이미 렌더링된 HTML과 CSS는 유지한다.

### 4.4 인증 화면의 CSS 요청 단계 축소

내용이 없는 `index.css`와 `login.css`의 `@import` 연결을 거치지 않고 두 인증 HTML이 `auth.css`를 직접 참조하게 한다. `common.css`와 `auth.css` 두 요청만으로 첫 화면 스타일을 완성한다.

### 4.5 화면별 초기 API 요청 최적화

화면마다 초기화 함수의 의존 관계를 명시하고, 독립 요청만 `Promise.all`로 동시에 실행한다.

- 메인 화면: 폴더를 먼저 확보한 뒤 활동 파일을 연결하는 흐름은 유지한다. 프로필 조회와 저장된 종합 분석 결과 조회는 폴더·파일 흐름과 동시에 시작한다.
- 파일 관리 화면: 폴더가 필요한 단계는 먼저 수행하고, 이후 서로 독립적인 활동 파일 조회와 프로젝트 분석 조회는 동시에 실행한다.
- 마이페이지·포트폴리오·활동 추천 화면: 화면 골격과 로컬 캐시를 먼저 표시하고 서버 조회가 끝나면 최신 데이터로 갱신한다. 같은 URL의 중복 GET 요청은 기존 세션 캐시를 공유한다.
- POST, PUT, PATCH, DELETE 및 파일 미리보기처럼 최신성이 필수인 요청에는 캐시를 추가하지 않는다.
- 인증 실패와 네트워크 실패 시 기존 로그인 안내, 로컬 저장 데이터 또는 빈 상태를 사용한다.

API 병렬화는 서버 호출 수를 늘리지 않고 기다리는 순서만 줄인다. 데이터 간 선후 관계가 있는 요청은 병렬화하지 않는다.

### 4.6 이미지와 외부 렌더링 차단 자원 개선

- 메인 튜토리얼의 PNG 이미지에 `loading="lazy"`와 `decoding="async"`를 적용하고 크기 정보를 제공하여 튜토리얼을 열기 전의 불필요한 다운로드와 화면 흔들림을 줄인다.
- 활동 추천 화면의 외부 Font Awesome CSS를 제거하고 이전·다음 화살표를 HTML 문자 또는 기존 CSS로 표현한다.
- 활동 추천 화면만 별도로 불러오던 외부 Pretendard CSS를 제거하고 공통 시스템 글꼴 목록을 사용한다. 외부 CDN 장애가 첫 화면 표시를 막지 않게 한다.
- 사용자 프로필 사진과 동적으로 생성되는 미리보기는 기능상 즉시 필요한 경우가 있으므로 일괄 지연 로딩하지 않고 화면별 필요성을 따로 판단한다.

## 5. 데이터 흐름

1. 브라우저가 정적으로 생성된 페이지 HTML을 Vercel CDN에서 받는다.
2. 정적으로 생성된 현재 화면의 CSS를 CDN에서 병렬로 받으며 화면 골격을 표시한다.
3. Next.js 클라이언트 코드가 준비되면 현재 화면에 필요한 레거시 스크립트 다운로드를 동시에 시작한다.
4. 준비된 스크립트는 선언 순서대로 실행한다.
5. 화면 초기화는 로컬 캐시로 즉시 표시할 수 있는 내용을 먼저 렌더링한다.
6. 선후 관계가 없는 읽기 API 요청을 동시에 실행하고, 도착한 데이터부터 해당 화면 영역을 갱신한다.
7. 사용자가 내부 링크에 이동 의도를 보일 때만 다음 화면을 미리 불러온다.

## 6. 오류 처리

- 정적 자원 경로가 허용 목록에 없거나 파일이 없으면 404를 반환한다.
- 스크립트 다운로드 오류는 콘솔에 경로와 함께 기록하고, 서버 렌더링된 화면은 유지한다.
- 선택적 API 요청 실패는 다른 독립 요청의 성공 결과를 취소하지 않는다.
- 인증이 필요한 데이터 요청이 401을 반환하면 기존 로그인 필요 안내를 유지한다.
- 로컬 캐시가 손상되었거나 사용할 수 없으면 캐시 없이 서버 요청을 수행한다.
- 병렬화 이후에도 데이터 생성·수정·삭제 성공 시 관련 읽기 캐시를 무효화한다.

## 7. 검증 계획

### 7.1 자동 검증

- 정적 구조 테스트에 다음 기준을 추가한다.
  - CSS·JavaScript Route Handler가 정적 생성 설정과 파일 목록을 가진다.
  - 전체 경로 일괄 사전 로딩 코드가 존재하지 않는다.
  - 레거시 스크립트가 `no-store`를 사용하지 않고 다운로드를 병렬 시작한다.
  - 인증 HTML이 `auth.css`를 직접 참조한다.
  - 튜토리얼 PNG가 지연 로딩과 비동기 디코딩을 사용한다.
  - 활동 추천 화면이 외부 렌더링 차단 CSS를 사용하지 않는다.
- `npm test`를 실행한다.
- `npm run build`를 실행하고 `/css/[...file]`, `/js/[...file]`이 정적 경로로 생성되는지 확인한다.

### 7.2 실행 검증

- 로컬 프로덕션 서버에서 `/`, `/main.html` 및 각 주요 화면이 정상 응답하는지 확인한다.
- 첫 요청과 반복 요청의 HTML, CSS, JavaScript 응답 시간과 캐시 헤더를 변경 전 측정값과 비교한다.
- 브라우저 네트워크 기록에서 첫 진입 직후 방문하지 않은 11개 화면 요청이 발생하지 않는지 확인한다.
- 로그인 버튼, 화면 이동, 공유 내비게이션, 프로필 표시, 파일 목록, 포트폴리오 목록, 활동 추천의 기본 동작을 확인한다.
- 느린 네트워크 또는 일부 API 실패 상황에서도 화면 골격과 다음 행동 안내가 표시되는지 확인한다.

## 8. 완료 기준

다음 조건을 모두 만족하면 이번 성능 개선을 완료한 것으로 본다.

1. 로그인 화면과 메인 화면을 포함한 모든 기존 페이지가 동일한 경로에서 열린다.
2. CSS와 JavaScript가 Vercel 서버 함수가 아닌 정적 생성 결과로 제공된다.
3. 첫 진입 시 사용하지 않는 전체 페이지 사전 로딩이 발생하지 않는다.
4. 현재 페이지 스크립트 다운로드가 병렬로 시작되고 선언 순서대로 실행된다.
5. 안전하게 분리 가능한 초기 읽기 요청이 병렬 실행된다.
6. 숨겨진 튜토리얼 이미지와 외부 렌더링 차단 자원이 초기 표시를 지연하지 않는다.
7. 자동 테스트와 프로덕션 빌드가 통과한다.
8. 변경 전보다 초기 네트워크 요청 수 또는 응답 대기 시간이 감소했음을 실행 측정으로 확인한다.

## 9. 제외 범위

- Supabase 스키마와 RLS 정책 변경
- OpenAI 분석·포트폴리오 생성 모델의 응답 시간 개선
- 포트폴리오 PDF/PPTX 변환 서버의 처리 시간 개선
- 기존 화면을 새로운 React 컴포넌트로 전면 재작성하는 작업
- 디자인이나 사용자 흐름의 기능적 변경
