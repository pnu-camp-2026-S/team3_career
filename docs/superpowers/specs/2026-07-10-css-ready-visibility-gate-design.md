# CSS 준비 전 HTML 노출 방지 설계

## 1. 목적

페이지 최초 진입과 내부 화면 이동에서 CSS가 적용되기 전에 HTML이 잠깐 보이는 현상을 제거한다. 사용자가 선택한 방식에 따라 별도 로딩 문구는 표시하지 않고, 현재 페이지에 필요한 CSS가 준비될 때까지 화면 내용만 보이지 않게 한다.

## 2. 현재 원인

현재 레거시 페이지 Route는 CSS 다운로드를 빠르게 시작할 수 있도록 `<head>`에 preload를 생성하지만, 실제 `<link rel="stylesheet">`는 페이지 본문 앞에 렌더링된다. 이 링크에는 React의 스타일시트 대기 동작을 활성화하는 설정이 없으므로, 특히 소프트 페이지 이동에서 새 HTML이 CSS 적용보다 먼저 화면에 반영될 수 있다.

## 3. 선택한 접근

`LegacyStyleGate` 클라이언트 컴포넌트를 추가하여 현재 페이지 HTML을 감싼다. 이 컴포넌트는 서버 렌더링 시 내용을 `visibility: hidden`으로 숨기고, 현재 페이지의 공통 CSS와 화면 CSS가 모두 준비되면 `visibility: visible`로 전환한다.

`display: none`을 사용하지 않는 이유는 숨겨진 동안에도 최종 화면 공간을 계산하게 하여 공개 시 레이아웃이 크게 이동하는 것을 줄이기 위함이다.

## 4. 구성 요소

### 4.1 스타일 링크 식별

`app/[[...slug]]/page.js`가 렌더링하는 각 스타일 링크에 현재 HTML 파일명을 나타내는 `data-legacy-style-page` 속성을 추가한다. 스타일 게이트는 이 값으로 이전 페이지의 링크와 현재 페이지의 링크를 구분한다.

### 4.2 LegacyStyleGate

새 파일 `app/LegacyStyleGate.jsx`는 다음 책임만 가진다.

1. 서버 렌더링과 첫 클라이언트 렌더링에서 자식 내용을 숨긴다.
2. 현재 페이지의 스타일 링크를 찾는다.
3. 각 링크의 `sheet` 속성으로 이미 로드된 CSS를 즉시 판별한다.
4. 아직 준비되지 않은 링크에는 `load`와 `error` 이벤트를 등록한다.
5. 링크가 아직 DOM에 없다면 `MutationObserver`로 링크 삽입을 감시한다.
6. 모든 링크가 성공 또는 실패 상태로 정리되면 내용을 공개한다.
7. 최대 3초가 지나면 영구 빈 화면을 막기 위해 내용을 공개한다.
8. 페이지가 바뀌거나 컴포넌트가 해제되면 이벤트, 감시자와 타이머를 정리한다.

컴포넌트는 `pageKey`와 예상 스타일 경로 목록을 입력으로 받으며, 내부 CSS 내용이나 페이지 기능에는 관여하지 않는다.

### 4.3 페이지 연결

`app/[[...slug]]/page.js`에서 실제 레거시 HTML과 `LegacyScripts`를 `LegacyStyleGate`로 감싼다. 컴포넌트에 페이지 파일명을 React `key`로 전달하여 소프트 이동 때마다 준비 상태가 다시 숨김 상태에서 시작되게 한다.

레거시 스크립트는 화면이 숨겨진 동안에도 기존처럼 초기화할 수 있다. CSS 준비 후 화면을 공개할 때 이미 초기 데이터가 반영되어 있을 가능성이 높아져 중간 상태 노출도 줄어든다.

### 4.4 JavaScript 비활성화 대응

JavaScript 자체가 실행되지 않으면 상태 전환과 3초 타이머도 동작할 수 없다. 따라서 `<noscript>` 안의 최소 CSS로 스타일 게이트의 `visibility`를 강제로 복원한다. 이 경우 CSS 적용 전 노출 가능성보다 페이지를 영구적으로 볼 수 없는 문제가 더 크므로 콘텐츠 접근을 우선한다.

## 5. 상태 흐름

1. 서버가 CSS 링크와 숨겨진 페이지 내용을 함께 응답한다.
2. 브라우저가 preload와 stylesheet 링크를 통해 CSS 다운로드를 시작한다.
3. React가 `useLayoutEffect`에서 현재 페이지의 스타일 링크를 확인한다.
4. 모든 CSS가 이미 준비됐으면 첫 화면 그리기 전에 바로 공개한다.
5. 준비되지 않은 CSS가 있으면 빈 화면을 유지하고 링크 이벤트를 기다린다.
6. 모든 링크가 정리되거나 3초가 지나면 페이지를 공개한다.
7. 내부 페이지 이동 시 새 `pageKey`의 게이트가 다시 같은 과정을 수행한다.

## 6. 오류 처리

- CSS 하나가 404 또는 네트워크 오류를 반환해도 다른 CSS가 정리될 때까지 기다린 뒤 화면을 공개한다.
- `load` 또는 `error` 이벤트가 누락되어도 3초 타임아웃으로 화면을 공개한다.
- 예상한 스타일 링크를 즉시 찾지 못하면 DOM 변경을 감시하여 링크가 추가되는 즉시 로딩 상태를 확인한다.
- 페이지 이동 중 이전 게이트가 해제되면 남은 이벤트 리스너, DOM 감시자와 타이머를 제거하여 이전 페이지가 새 페이지 상태를 변경하지 못하게 한다.
- 기존 CSS 캐시 정책, 데이터 API, 인증, 페이지 스크립트 실행 순서는 변경하지 않는다.

## 7. 테스트 계획

정적 구조 테스트에 다음 기준을 추가한다.

- `LegacyStyleGate`가 서버와 첫 렌더링에서 `visibility: hidden`을 사용한다.
- `useLayoutEffect`에서 스타일 링크의 `sheet`, `load`, `error` 상태를 확인한다.
- 늦게 추가되는 스타일 링크를 `MutationObserver`로 감시한다.
- 3초 실패 복구 타이머와 정리 함수가 존재한다.
- 페이지 Route가 스타일 링크에 현재 페이지 식별자를 넣는다.
- 레거시 HTML과 스크립트가 페이지별 `key`를 가진 스타일 게이트 내부에 있다.
- JavaScript 비활성화 시 콘텐츠를 공개하는 `<noscript>` 처리가 존재한다.

구현 후 다음 명령을 실행한다.

```bash
npm test
npm run build
```

프로덕션 빌드 결과 HTML에서 CSS preload와 stylesheet 링크는 유지되고, 실제 페이지 내용의 상위 요소가 `visibility: hidden`으로 시작하는지 확인한다. 가능한 브라우저 환경에서는 느린 네트워크에서 스타일 없는 HTML이 노출되지 않는지도 확인한다.

## 8. 완료 기준

1. 최초 진입과 소프트 페이지 이동 모두 CSS 준비 전에는 페이지 HTML이 보이지 않는다.
2. CSS가 준비되면 별도 로딩 문구 없이 페이지가 즉시 표시된다.
3. CSS 오류 또는 이벤트 누락 시에도 최대 3초 후에는 페이지를 볼 수 있다.
4. JavaScript 비활성화 시 콘텐츠가 영구적으로 숨겨지지 않는다.
5. 기존 페이지 경로, CSS 순서, 레거시 스크립트, 데이터 요청 동작이 유지된다.
6. 자동 테스트와 프로덕션 빌드가 통과한다.

## 9. 제외 범위

- 별도 로딩 화면 또는 스켈레톤 UI 추가
- CSS 파일 내용과 화면 디자인 변경
- 기존 HTML 화면을 React 컴포넌트로 전면 재작성
- Vercel CDN 및 Supabase 설정 변경
