# 포트폴리오 PDF 변환기 환경변수 로딩 설계

## 목적

로컬 개발 서버의 `/api/portfolio/render-pdf`가 프로젝트 루트의 `key.env`에 등록된 PDF 변환기 설정을 읽도록 한다. Vercel 등 배포 환경에서는 플랫폼에 등록된 환경변수를 그대로 우선 사용한다.

## 접근 방식 비교

1. `key.env`만 강제로 로드하면 구현은 단순하지만 배포 환경변수를 파일 값이 덮어쓸 위험이 있다.
2. 모든 설정을 `.env.local`로 옮기면 Next.js 기본 방식과 맞지만 현재 프로젝트가 공통으로 사용하는 `key.env` 구조를 바꿔야 한다.
3. 배포 환경변수를 우선하고 비어 있는 값만 `key.env`에서 보충하면 현재 구조와 배포 안전성을 함께 유지할 수 있다.

따라서 세 번째 방식을 사용한다.

## 구현 설계

- `app/api/portfolio/render-pdf/route.js`에서 `dotenv`와 `path`를 사용해 프로젝트 루트의 `key.env`를 읽는다.
- `dotenv.config`의 기본 `override: false` 동작을 사용한다. 이미 프로세스에 등록된 Vercel 환경변수는 덮어쓰지 않는다.
- 기존 URL 형식 검사, 변환 제한 시간, 선택적 Bearer 토큰, 503 폴백 동작은 변경하지 않는다.
- 환경변수의 값이나 토큰은 로그와 테스트 출력에 노출하지 않는다.

## 데이터 흐름

1. 서버가 PDF 변환 API 모듈을 불러온다.
2. 운영체제 또는 Vercel 환경변수를 먼저 유지한 채 `key.env`의 누락된 값을 보충한다.
3. `PORTFOLIO_CONVERTER_URL`로 PPTX 변환 워커에 요청한다.
4. 성공하면 PDF를 반환하고, 설정이 없거나 워커를 사용할 수 없으면 기존 캔버스 미리보기로 전환한다.

## 오류 처리

- `key.env`가 없어도 서버 시작을 막지 않는다. 배포 환경변수만으로 동작할 수 있어야 한다.
- URL이 HTTP 또는 HTTPS 형식이 아니면 기존 검증 오류를 유지한다.
- 토큰은 선택값이지만 설정된 경우 기존처럼 `Authorization` 헤더에만 사용한다.

## 검증 기준

- 정적 구조 테스트에서 PDF 변환 라우트가 `key.env`를 로드하고 있음을 확인한다.
- `npm test`와 `npm run build`를 실행한다.
- 실제 환경변수 값은 테스트나 명령 출력에 표시하지 않는다.
