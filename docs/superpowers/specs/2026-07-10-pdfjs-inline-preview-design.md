# PDF.js 화면 내 미리보기 설계

## 목적

브라우저 내장 PDF 뷰어가 `blob:` PDF를 iframe 안에서 빈 화면으로 표시하는 문제를 제거한다. 변환된 실제 PDF의 각 페이지를 포트폴리오 생성 화면 안에서 동일하게 확인할 수 있어야 한다.

## 접근 방식 비교

1. `<object>` 또는 `<embed>`로 바꾸는 방식은 수정량이 작지만 브라우저 내장 PDF 기능에 계속 의존하므로 같은 문제가 반복될 수 있다.
2. PDF.js를 브라우저에서 실행하는 방식은 실제 PDF 페이지를 캔버스에 렌더링하므로 브라우저의 PDF 플러그인 설정과 무관하게 동작한다.
3. 서버에서 PDF를 PNG로 변환하는 방식은 화면 표시가 안정적이지만 서버 부하와 API 복잡도가 커진다.

따라서 두 번째 방식을 사용한다.

## 의존성과 자산 관리

- `pdfjs-dist`를 프로젝트의 직접 의존성으로 등록한다.
- 개발 및 빌드 전에 복사 스크립트가 PDF.js 본체와 워커 파일을 `public/vendor/pdfjs`로 복사한다.
- 생성된 벤더 파일은 Git에 직접 저장하지 않고 원본 패키지와 복사 스크립트만 관리한다.
- 외부 CDN에 의존하지 않아 로컬 개발과 배포 환경에서 같은 버전을 사용한다.

## 화면 렌더링

- `/api/portfolio/render-pdf`가 반환한 PDF Blob을 `ArrayBuffer`로 읽는다.
- PDF.js로 문서를 열고 모든 페이지를 순서대로 캔버스에 렌더링한다.
- 페이지는 기존 미리보기 영역 안에서 세로로 배치하고, 영역 높이를 넘으면 내부 스크롤로 확인한다.
- 각 페이지에는 `PDF 1페이지`처럼 접근 가능한 설명을 제공한다.
- PDF 객체 URL은 다운로드 또는 새 창 열기 같은 별도 기능에 필요하지 않으므로 만들지 않는다.

## 상태와 오류 처리

- 변환 중에는 기존 로딩 상태를 유지한다.
- PDF.js 로딩이나 PDF 페이지 렌더링이 실패하면 원인을 설명하는 한국어 오류 상태와 `다시 시도` 버튼을 표시한다.
- 변환 API가 503을 반환할 때만 기존 PPTX 매칭 캔버스 폴백을 사용한다.
- 이전 렌더링 요청은 기존 `AbortController`와 요청 번호로 취소하고, 오래된 PDF 렌더링 결과가 새 초안을 덮어쓰지 않게 한다.

## 테스트와 검증

- 정적 구조 테스트에서 iframe이 제거되고 PDF.js 로딩, 페이지 반복 렌더링, 오류 상태가 포함됐는지 확인한다.
- 실제 변환 PDF를 PDF.js로 렌더링해 페이지 캔버스가 생성되는지 확인한다.
- `npm test`와 `npm run build`를 모두 통과해야 한다.
- 기존 PPTX 다운로드, 포트폴리오 수정, 캔버스 폴백 동작은 유지한다.
