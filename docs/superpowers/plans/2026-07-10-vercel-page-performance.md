# Vercel 페이지 로딩 성능 개선 구현 계획

## 목표

승인된 설계에 따라 모든 기존 화면의 경로와 기능을 유지하면서 정적 자산 전달, 초기 네트워크 요청, 스크립트 로딩, 이미지 및 외부 의존성을 최적화한다.

## 작업 1. CSS·JavaScript Route Handler 정적 생성

대상 파일:

- `app/css/[...file]/route.js`
- `app/js/[...file]/route.js`
- `tests/static-html-structure.test.js`

구현 내용:

1. 각 원본 디렉터리의 파일을 열거하는 `generateStaticParams`를 추가한다.
2. `force-dynamic`을 `force-static`으로 변경하고 등록되지 않은 동적 경로는 404가 되게 한다.
3. 요청 헤더와 파일 수정 시각에 의존하는 ETag 분기를 제거한다.
4. Vercel CDN 공유 캐시를 허용하는 응답 헤더를 설정한다.
5. 경로 이탈 방지와 파일 확장자별 콘텐츠 유형 처리는 유지한다.
6. 구조 테스트에서 정적 생성 설정과 경로 안전성을 검증한다.

검증:

```bash
npm test
npm run build
```

빌드 경로 표에서 `/css/[...file]`, `/js/[...file]`이 정적으로 생성되는지 확인한다.

## 작업 2. 내비게이션과 레거시 스크립트 로딩 최적화

대상 파일:

- `app/LegacyNavigation.jsx`
- `app/LegacyScripts.jsx`
- `tests/static-html-structure.test.js`

구현 내용:

1. 마운트 직후 전체 경로를 사전 로딩하는 반복문을 제거한다.
2. 내부 링크의 `pointerenter`와 `focusin` 이벤트에서 해당 경로만 사전 로딩한다.
3. 이미 사전 로딩한 경로는 집합으로 관리하여 중복 요청하지 않는다.
4. 모든 외부 스크립트의 다운로드를 동시에 시작한다.
5. 다운로드가 끝난 내용은 HTML 선언 순서대로 실행한다.
6. `no-store`를 제거하고 기존 페이지 전환 취소·전역 리스너 정리 동작은 유지한다.
7. 구조 테스트에서 전체 사전 로딩 금지, 의도 기반 로딩, 병렬 다운로드와 순서 보존을 검증한다.

검증:

```bash
npm test
```

## 작업 3. 렌더링 차단 CSS와 초기 이미지 요청 축소

대상 파일:

- `html/index.html`
- `html/login.html`
- `html/main.html`
- `html/contest.html`
- `css/contest.css`
- `tests/static-html-structure.test.js`

구현 내용:

1. 인증 화면이 빈 중간 CSS 대신 `auth.css`를 직접 불러오게 한다.
2. 메인 튜토리얼 PNG에 지연 로딩, 비동기 디코딩, 실제 크기 정보를 추가한다.
3. 활동 추천 화면의 외부 Pretendard와 Font Awesome CSS를 제거한다.
4. 달력 이전·다음 아이콘을 외부 글꼴 없이 표현하고 기존 접근성 이름을 유지한다.
5. 구조 테스트의 화면별 CSS 기대값을 실제 공용 인증 CSS 구조에 맞게 수정한다.

검증:

```bash
npm test
```

## 작업 4. 초기 읽기 API 의존 관계 최적화

대상 파일:

- `js/main.js`
- `js/create.js`
- 필요할 경우 초기 읽기 요청이 있는 다른 `js/*.js`
- `tests/static-html-structure.test.js`

구현 내용:

1. 각 초기화 함수에서 서버 요청의 입력·출력 의존성을 다시 확인한다.
2. 메인 화면의 폴더→활동 파일 흐름은 유지하면서 프로필과 저장된 종합 분석 결과 요청을 동시에 시작한다.
3. 파일 관리 화면은 폴더 확보 후 활동 파일과 폴더별 분석 조회가 서로 독립적일 때 동시에 실행한다.
4. 다른 화면은 독립 요청이 명확할 때만 병렬화한다. 요청 수를 늘리거나 데이터 선후 관계를 바꾸는 수정은 하지 않는다.
5. 일부 읽기 요청 실패가 다른 성공 결과를 취소하지 않도록 각 함수의 기존 오류 처리를 유지한다.
6. 인증 확인, 쓰기 요청, 미리보기, AI 생성 요청의 캐시 정책은 변경하지 않는다.

검증:

```bash
npm test
```

로그인하지 않은 상태와 API 일부 실패 상태에서도 기존 빈 화면 또는 로그인 안내가 유지되는지 확인한다.

## 작업 5. 전체 빌드와 실행 성능 비교

대상 파일:

- 필요한 경우 성능 회귀 검증을 위한 `tests/static-html-structure.test.js`

검증 내용:

1. 전체 자동 테스트와 프로덕션 빌드를 실행한다.
2. 로컬 프로덕션 서버에서 모든 기존 페이지와 정적 자산의 HTTP 상태를 확인한다.
3. `/`, `/main.html`, 대표 CSS·JavaScript의 첫 요청과 반복 요청 시간을 측정한다.
4. 캐시 헤더와 빌드 출력으로 서버 함수 호출이 제거됐음을 확인한다.
5. 브라우저 네트워크 기록으로 첫 화면에서 방문하지 않은 전체 경로 요청이 발생하지 않는지 확인한다.
6. 로그인 버튼, 내부 화면 이동, 공유 내비게이션, 메인 데이터 표시, 파일 관리, 활동 추천, 포트폴리오 목록의 기본 동작을 점검한다.

최종 명령:

```bash
npm test
npm run build
```

## 완료 조건

- 승인된 설계 문서의 8개 완료 기준을 모두 충족한다.
- 프로덕션 빌드와 테스트가 통과한다.
- 변경 전보다 불필요한 초기 요청 수가 줄고 CSS·JavaScript가 정적 응답으로 제공되는 근거가 남는다.
- 기존 페이지 경로와 주요 기능에 회귀가 없다.
