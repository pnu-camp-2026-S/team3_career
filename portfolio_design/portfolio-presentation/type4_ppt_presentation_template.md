# 4번 PPT 발표형 템플릿

## 1. 템플릿 목적

이 템플릿은 **프로젝트 하나**를 짧은 발표 흐름에 맞춰 설명하는 **3장짜리 PPT 발표 자료**를 만들기 위한 Markdown 입력 템플릿이다.

상세 기술 설명보다 발표자가 말로 보충할 수 있는 핵심 정보만 남기며, 전체 흐름은 다음 순서로 구성한다.

```text
1. 프로젝트 개요
2. 진행 과정
3. 결과 및 결론
```

---

## 2. 공통 디자인 기준

- 출력 형식: **16:9 가로형 PPT**
- 배경: **흰색**
- 포인트 색상: **진한 남색**
- 상단 요소: **얇은 남색 라인**
- 제목 위치: **왼쪽 상단**
- 보조 제목: 영문 소제목을 작게 표시 가능
- 내용 영역: **연한 라벤더/블루 톤의 큰 박스** 사용
- 장식 요소: 아이콘, 사진, 차트, 카드형 장식은 기본적으로 제외
- 텍스트 분량: 발표자가 말로 보충할 수 있도록 짧게 작성
- 구성 원칙: **한 장에는 하나의 역할만 담기**

---

## 3. 프로젝트 기본 정보 입력

```text
프로젝트명: [프로젝트명을 입력하세요]
프로젝트 한 줄 요약: [프로젝트를 1~2문장으로 요약하세요]
지원 직무 또는 발표 목적: [선택 입력]
사용한 자료 출처: [Wiki / 활동 기록 / 업로드 자료 등 승인된 근거를 입력하세요]
```

# Slide 1. 프로젝트 개요

## Project Overview

### 슬라이드 역할

문제, 목표, 역할을 한 장에서 간단히 설명한다.

### 레이아웃

```text
상단 얇은 남색 라인

[왼쪽 상단]
프로젝트 개요
Project Overview

[오른쪽 상단]
[프로젝트를 관통하는 짧은 안내 문장 또는 요약 문장]

[중앙 넓은 요약 박스]
프로젝트명
프로젝트 한 줄 요약

[하단 3개 영역]
문제 | 목표 | 역할
```

### 입력 내용

#### 오른쪽 상단 안내 문장

```text
[프로젝트의 문제, 목표, 역할을 압축한 한 문장을 입력하세요]
```

#### 중앙 요약 박스

```text
프로젝트명: [입력]
프로젝트 한 줄 요약: [어떤 문제를 어떤 방향으로 해결했는지 1~2문장으로 입력]
```

#### 하단 3요소

| 항목 | 내용 |
|---|---|
| 문제 | [해결하려 한 문제를 짧게 입력] |
| 목표 | [프로젝트 목표를 1~2문장으로 입력] |
| 역할 | [본인이 맡은 역할만 간단히 입력] |

### 작성 기준

- 프로젝트명을 임의로 만들지 않는다.
- 역할은 본인이 실제 수행한 내용만 작성한다.
- 문제, 목표, 역할은 각각 1~2문장 이내로 압축한다.

# Slide 2. 진행 과정

## Process Page

### 슬라이드 역할

요구 분석부터 수정까지의 흐름과 진행 중 어려움, 해결 방식을 간단히 정리한다.

### 레이아웃

```text
상단 얇은 남색 라인

[왼쪽 상단]
진행 과정
Process Page

[중앙 흐름 영역]
요구 분석 → 화면 설계 → 구현 → 수정

[하단 정리 영역]
어려움: [한 줄 정리]
해결: [한 줄 정리]
```

### 입력 내용

#### 중앙 흐름

```text
요구 분석 → 화면 설계 → 구현 → 수정
```

| 단계 | 내용 |
|---|---|
| 요구 분석 | [요구 사항, 문제 상황, 사용자 니즈 등을 짧게 입력] |
| 화면 설계 | [화면 구조, 사용자 흐름, 설계 방향 등을 짧게 입력] |
| 구현 | [핵심 구현 내용 또는 맡은 작업을 짧게 입력] |
| 수정 | [피드백 반영, 개선, 보완 내용을 짧게 입력] |

#### 하단 정리

```text
어려움: [진행 중 마주한 제약이나 문제를 한 줄로 입력]
해결: [어떻게 판단하고 조정했는지 한 줄로 입력]
```

### 작성 기준

- 복잡한 상세 설명은 넣지 않는다.
- 발표자가 말로 보충할 수 있도록 핵심 키워드 중심으로 작성한다.
- 어려움과 해결은 과장하지 않고 실제 경험 기반으로 작성한다.

# Slide 3. 결과 및 결론

## Result Conclusion

### 슬라이드 역할

구현 결과, 프로젝트의 의미, 개선 방향, 배운 점을 정리한다.

### 레이아웃

```text
상단 얇은 남색 라인

[왼쪽 상단]
결과 및 결론
Result Conclusion

[상단 또는 중간]
구현 결과 한 문장

[하단 절반 큰 연한 배경 박스]
왼쪽: 결론 문장 크게 배치
오른쪽: 결과 / 의미 / 개선 짧은 문단 정리
```

### 입력 내용

#### 구현 결과 한 문장

```text
[프로젝트를 통해 구현하거나 정리한 결과를 한 문장으로 입력]
```

#### 하단 큰 박스

```text
결론 문장: [프로젝트를 통해 확인한 핵심 결론을 짧고 명확하게 입력]
배운 점: [프로젝트를 통해 배운 점을 1~2문장으로 입력]
```

#### 오른쪽 정리

| 항목 | 내용 |
|---|---|
| 결과 | [구현 결과 또는 산출물을 짧게 입력] |
| 의미 | [프로젝트의 의미, 배운 점, 직무와의 연결점을 입력] |
| 개선 | [추가 개선 방향 또는 보완할 점을 입력] |

### 작성 기준

- 없는 성과 수치나 결과를 만들지 않는다.
- 구현 결과와 배운 점을 구분해서 작성한다.
- 개선 방향은 실현 가능한 범위로 작성한다.

---

## 4. AI 작성 원칙

- 프로젝트 하나를 **3장 안에 설명할 수 있도록 압축**한다.
- 승인된 Wiki, 활동 기록, 업로드 자료에 있는 내용만 사용한다.
- 없는 성과, 수치, 역할, 기술, 회사명은 만들지 않는다.
- 발표자가 말로 보충할 수 있도록 문장은 짧게 작성한다.
- 상세한 기술 설명보다 **개요 → 과정 → 결과** 흐름을 우선한다.
- 이미지나 차트는 기본적으로 사용하지 않는다.
- 사용자가 이미지나 차트를 명시적으로 요청한 경우에만 추가한다.

---

## 5. 피해야 할 것

- 프로젝트 여러 개를 한 번에 넣는 것
- 1번 상세기술 포트폴리오처럼 페이지를 계속 늘리는 것
- 카드, 아이콘, 사진, 차트를 과하게 넣는 것
- 한 페이지에 많은 문장을 넣어 발표 여백을 없애는 것
- 근거 없는 성과를 발표용으로 과장하는 것
- 본인이 하지 않은 역할이나 사용하지 않은 기술을 넣는 것
